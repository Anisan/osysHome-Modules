# Modules - руководство пользователя

## 1. Для чего нужен модуль

`Modules` нужен для централизованного управления модулями и обновлениями в `osysHome`.

С его помощью пользователь может:

- видеть список уже установленных модулей;
- находить доступные модули из GitHub;
- устанавливать новые модули;
- обновлять отдельные модули и само ядро `osysHome`;
- открывать настройки установленных модулей;
- управлять фоновыми циклами модулей, если они поддерживаются;
- удалять модули, которые не относятся к защищённым системным.

По сути, `Modules` является точкой входа для сопровождения экосистемы модулей внутри системы.

## 2. Что пользователь видит в интерфейсе

В административном интерфейсе `Modules` пользователь видит:

- кнопку `Refresh` для обновления данных;
- переключатель `Only with updates`, чтобы оставить только модули с доступными обновлениями;
- строку `Filter...` для поиска по карточкам;
- кнопку `Settings` для настройки самого модуля `Modules`;
- вкладку `Installed` со списком установленных модулей;
- вкладку `Available` со списком модулей, найденных в GitHub;
- карточки модулей с краткой информацией и действиями;
- модальное окно подробной информации о выбранном модуле.

### Что отображается в карточке установленного модуля

На карточке установленного модуля обычно видны:

- иконка модуля;
- имя или заголовок;
- описание;
- автор;
- версия;
- метки действий, например `search`, `cycle`, `widget`;
- ссылка на GitHub-репозиторий, если он известен;
- ссылка на документацию, если она индексируется `Docs`;
- кнопка `Upgrade`, если найдено обновление;
- кнопки `Start cycle`, `Stop cycle`, `Restart cycle`, если модуль поддерживает циклическую задачу.

Если модуль отключён, карточка отображается приглушённо и помечается как `Inactive`.

### Что отображается в карточке доступного модуля

На вкладке `Available` пользователь видит:

- имя репозитория;
- описание репозитория;
- автора;
- темы GitHub, если они заданы;
- количество звёзд;
- кнопку `Install`.

## 3. Откуда модуль получает информацию

`Modules` объединяет данные из нескольких источников.

### Из самой системы `osysHome`

Из API и базы данных модуль получает:

- список установленных модулей;
- их текущие настройки;
- признаки активности;
- версию, описание, категорию, URL репозитория и дату последнего обновления;
- статус фонового цикла, если он есть;
- системные признаки обновления ядра.

### Из GitHub

Из GitHub модуль получает:

- список репозиториев по поиску `osysHome`;
- описание репозиториев;
- владельца репозитория;
- ветку по умолчанию;
- дату последних коммитов;
- список коммитов для выбранного модуля;
- количество звёзд;
- изображения модулей из `static/<ModuleName>.png`, если файл есть в репозитории.

### Из конфигурации самого `Modules`

Сам модуль использует свои настройки:

- `update_time` - как часто проверять обновления;
- `token` - GitHub token для запросов к API и увеличения лимитов.

> [!NOTE]
> Без токена GitHub модуль тоже работает, но при большом количестве запросов можно быстрее упереться в лимиты API.

## 4. Как устроена работа с модулями

Обычный сценарий пользователя выглядит так:

```mermaid
flowchart LR
    A[Пользователь открывает Modules] --> B[Modules читает установленные модули]
    B --> C[Modules запрашивает GitHub]
    C --> D[Показывает Installed и Available]
    D --> E[Пользователь открывает карточку модуля]
    E --> F[Просмотр истории, настроек, обновления или установки]
```

## 5. Глобальные настройки самого Modules

Кнопка `Settings` в верхней части страницы открывает настройки самого модуля.

### Доступные параметры

| Поле | Что означает |
| --- | --- |
| `Update time` | Интервал проверки обновлений в минутах |
| `Token` | GitHub token для доступа к API |

### Когда менять эти настройки

- уменьшайте `Update time`, если нужно быстрее видеть новые обновления;
- увеличивайте `Update time`, если хотите снизить частоту запросов;
- заполняйте `Token`, если используете много модулей или часто открываете историю и список репозиториев.

## 6. Что можно делать с установленными модулями

При выборе установленного модуля открывается окно с подробной информацией.

### Вкладка History

Здесь пользователь видит:

- последние коммиты выбранной ветки;
- автора коммита;
- сообщение коммита;
- дату коммита;
- отметку `Installed` для текущей установленной версии;
- возможность установить конкретный коммит.

Это удобно, если нужно:

- понять, что изменилось в новой версии;
- обновить модуль не до последнего состояния, а до конкретного коммита;
- проверить, на какой версии сейчас работает система.

### Вкладка Settings для установленного модуля

Для большинства установленных модулей доступны следующие параметры:

| Поле | Назначение |
| --- | --- |
| `Active` | Включает или отключает модуль |
| `Title` | Пользовательское имя модуля в интерфейсе |
| `Category` | Категория модуля |
| `Hidden in statusbar` | Скрывает модуль в панели управления |
| `Hide widget in control panel` | Скрывает виджет модуля на панели |
| `Url repository for update` | Репозиторий, из которого модуль будет обновляться |
| `Branch repository for update` | Ветка, по которой искать коммиты и обновления |
| `Logging level` | Уровень логирования модуля |

После изменения этих параметров нужно нажать `Save`.

### Что означают настройки установленного модуля

`Active`:
временно отключает модуль без его удаления.

`Title`:
позволяет заменить техническое имя на более понятное для пользователя.

`Category`:
используется для логической группировки модулей.

`Hidden in statusbar`:
полезно, если модуль не нужно показывать среди обычных элементов управления.

`Hide widget in control panel`:
позволяет оставить модуль установленным, но убрать его виджет с панели.

`Url repository for update`:
нужен, если модуль должен обновляться не из автоматически найденного репозитория, а из явно заданного URL.

`Branch repository for update`:
помогает закрепить модуль на конкретной ветке, например `master`, `main` или тестовой ветке разработки.

`Logging level`:
используется для диагностики. Для обычной работы чаще всего подходит значение по умолчанию, а `DEBUG` удобно включать только при разборе ошибок.

## 7. Что можно настроить для самого osysHome

Для карточки `osysHome` в модальном окне доступны:

- история коммитов ядра;
- выбор ветки ядра для обновления;
- обновление ядра до последнего или конкретного коммита.

Основная настройка здесь одна:

| Поле | Назначение |
| --- | --- |
| `Branch repository for update` | Ветка репозитория `osysHome`, по которой модуль будет искать обновления ядра |

## 8. Установка нового модуля

На вкладке `Available` показаны модули, найденные в GitHub.

Чтобы установить модуль:

1. Откройте `Modules`.
2. Перейдите на вкладку `Available`.
3. Найдите нужный модуль через фильтр или список.
4. Нажмите `Install`.
5. Дождитесь завершения установки.
6. Если система запросит перезапуск, выполните его.

Во время установки модуль:

- определяет репозиторий;
- скачивает архив с GitHub;
- распаковывает файлы в папку `plugins/<ModuleName>`;
- при наличии `requirements.txt` устанавливает зависимости;
- регистрирует модуль в системе.

> [!IMPORTANT]
> Для автоматического обнаружения ожидается соглашение по имени репозитория: `osysHome-ModuleName`.

## 9. Обновление установленных модулей

Если для модуля найдено обновление, пользователь увидит кнопку `Upgrade`.

Есть два варианта обновления:

- быстрое обновление с карточки;
- выбор конкретного коммита через вкладку `History`.

После обновления:

- у модуля сбрасывается признак доступного обновления;
- в системе выставляется флаг необходимости перезапуска;
- при наличии `requirements.txt` зависимости могут быть обновлены автоматически.

## 10. Удаление модулей

Удаление доступно не для всех модулей.

Нельзя удалить:

- `Modules`;
- `Objects`;
- `Users`;
- `Scheduler`;
- `wsServer`.

Для остальных модулей удаление:

- удаляет файлы модуля;
- удаляет запись модуля из системы;
- пытается очистить связанные таблицы базы данных;
- требует подтверждения в интерфейсе.

## 11. Виджет и уведомления

`Modules` добавляет виджет в панель управления.

Виджет показывает:

- наличие обновления для ядра `osysHome`;
- список модулей, у которых есть обновления.

Также модуль создаёт уведомления, когда:

- найдено новое обновление;
- выполнена установка;
- завершено обновление;
- произошла ошибка установки, обновления или удаления.

## 12. Поиск и фильтрация

Пользователь может:

- вводить текст в `Filter...` для поиска по карточкам;
- включать `Only with updates`, чтобы оставить только проблемно-актуальные позиции;
- использовать поиск системы, если модуль подключён через действие `search`.

Фильтр запоминается в браузере и сохраняется между открытиями страницы.

## 13. Практические сценарии

### Когда удобно открыть Modules

- после установки новой версии `osysHome`;
- когда нужно проверить, есть ли обновления плагинов;
- когда нужно временно отключить модуль без удаления;
- когда надо изменить ветку обновления конкретного модуля;
- когда требуется установить новый модуль из GitHub;
- когда нужно посмотреть историю коммитов перед обновлением.

### Когда стоит настроить GitHub token

- если у вас много модулей;
- если вы часто открываете вкладки с историей;
- если система регулярно проверяет обновления;
- если без токена GitHub начинает ограничивать запросы.

## 14. Что важно помнить

- `Modules` показывает не только локальные данные, но и сведения из GitHub;
- настройки самого `Modules` влияют на проверку обновлений для всей системы;
- настройки внутри карточки модуля относятся уже к конкретному установленному модулю;
- для части операций после установки или обновления может потребоваться перезапуск системы;
- защищённые системные модули удалить нельзя.
